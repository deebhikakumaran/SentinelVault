# Security & Privacy 

When managing automated cross-chain DeFi positions, security cannot be an afterthought. SentinelVault leverages the unique capabilities of the Chainlink CRE ecosystem, specifically Trusted Execution Environments (TEEs), to ensure data integrity, API key confidentiality, and strictly controlled access.

---

## 1. Confidential Data Sourcing (Privacy Track)

To calculate the Market Sentiment score, the Risk Engine needs real-time price data from centralized APIs (like CryptoCompare). However, injecting a private API key into a transparent, distributed network normally exposes that key to the public.

**The Solution:** `ConfidentialHTTPClient`
SentinelVault uses Chainlink CRE's TEE capabilities. When the workflow runs:
1.  The workflow executes inside a secure enclave.
2.  The API key is injected at runtime using a Go template: `Apikey {{.marketDataApiKey}}`.
3.  The API key is stored securely in the VaultDON secrets and is **never** written into the workflow code, configuration files, or on-chain state.

**Why `encryptOutput: false`?**
The data returned by CryptoCompare (the price changes of BTC, ETH, AAVE, etc.) is public knowledge. There is no need to encrypt the response body. Our confidentiality guarantee is placed entirely on the *authorization credential* (the API key), ensuring no one can steal your key to drain your API quota.

## 2. Access Control and Authorization

The SentinelVault smart contracts are designed so that no malicious actor can force a liquidation or steal funds.

### The `onlyAuthorized` Modifier
Both `SentinelVault.sol` and `SentinelRegistry.sol` protect their `onReport()` functions using an `onlyAuthorized` modifier:

```solidity
function _onlyAuthorized() internal view {
    require(
        msg.sender == owner ||
        (forwarder != address(0) && msg.sender == forwarder),
        "SentinelVault: caller not authorized"
    );
}
```

*   **The Forwarder:** In a production environment, `forwarder` is set to the Chainlink `KeystoneForwarder`. This means the vault will *only* execute a rescue operation if it receives a cryptographically signed payload generated by the decentralized oracle network (DON).
*   **The Owner:** The owner (deployer) retains fallback access. This is strictly for initial setup, local testing, and emergency manual interventions.

## 3. MEV Protection

Because SentinelVault performs automated swaps on decentralized exchanges (Uniswap V3), it is theoretically vulnerable to MEV (Miner Extractable Value) sandwich attacks. If a bot sees SentinelVault attempting to swap a large amount of WETH for USDC, it could manipulate the pool price to steal value.

**The Solution:** Slippage Guards
Inside `SentinelVault.sol`, before interacting with the Swap Router, the contract calculates a strict `amountOutMinimum`.

```solidity
uint256 expectedOut = (amountIn * _getEthPriceUsd()) / 1e12;
uint256 minOut = (expectedOut * (10_000 - maxSlippageBps)) / 10_000;
```

1.  It gets a reliable, baseline price using the Chainlink ETH/USD `PRICE_FEED`.
2.  It calculates exactly how much USDC it *should* get based on that reliable feed.
3.  It subtracts a configurable `maxSlippageBps` (e.g., 500 basis points, or 5%).
4.  It passes this `minOut` to Uniswap. If an MEV bot has manipulated the pool beyond this 5% tolerance, the swap safely reverts.

---

## 4. Graceful Fallback (No-Revert Guarantee)

What happens if the swap reverts due to MEV protection, or if the DEX has zero liquidity (common on testnets)? 

In poorly designed systems, the entire transaction reverts, the execution fails, and the user's Health Factor remains dangerously low.

**The Solution:** Catch and Hold
SentinelVault wraps the swap call in a `try/catch` block. 
If `SWAP_ROUTER.exactInputSingle()` fails:
1.  The transaction does **not** revert.
2.  The contract emits a `SwapSkipped` event.
3.  The WETH that was withdrawn from Aave is safely held in the `SentinelVault` balance.
4.  During the next workflow tick (5 minutes later), the `_emergencyExit` function has a specific "Pass 1" rule: it checks if there is any loose WETH in the vault and attempts to swap and repay it *first* before withdrawing more collateral. 

This ensures that under adverse network conditions, your collateral is still moved out of Aave and into the safety of the vault.
